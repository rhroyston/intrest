What is intREST?

IntREST is intended to enable your Web browser to act as a standalone real time management platform for devices having a REST API. The code is completely client side; there is no backend server. IntREST utilizes Material Design Lite and Google Charts as the presentation foundation and relies on your browsers support of the XMLHttpRequest API for the HTTP transport.

How does it work?

Client side only. Your data and credentials are isolated to your browser and managed devices. JavaScript in your browser initiates communications to managed devices which respond with data which is then displayed by your browser. Your browsers view configuration is stored in local storage. For backup purposes you can download your intREST view configuration file and store it wherever you would like. There is no server required or used by intREST. The data fetched from managed devices is not stored.

What is REST?

Representational state, REST, is a communications architecture or style; the World Wide Web is an example of RESTful architecture.

Representational state transfer (REST) or RESTful Web services are one way of providing interoperability between computer systems on the Internet. REST-compliant Web services allow requesting systems to access and manipulate textual representations of Web resources using a uniform and predefined set of stateless operations. Other forms of Web service exist, which expose their own arbitrary sets of operations such as WSDL and SOAP. -Wikipedia

A subset of HTTP verbs and their behavior are primarily what one needs to know to utilize a RESTful API. DELETE, GET, POST, and PUT are methods that pass arguments to the REST API. The difference between POST and PUT is that POST creates while PUT updates or replaces.

What is CORS?

Cross Origin Resource Sharing allows scripts, namely JavaScript, access to content hosted on a domain other than itself. By default, Web resources disallow this functionality but Web servers can specify exceptions via a response header. Absent this exception header, scripts are disallowed from communicating with foreign Web resources; unless you install a plugin on your browser to bypass this basic security feature. CORS plugins are available for Chrome and Firefox and are required if you are using a REST API on a device that fails to set the Access-Control-Allow-Origin header. See the Troubleshooting section below.

A resource makes a cross-origin HTTP request when it requests a resource from a different domain than the one which the first resource itself serves. For example, an HTML page served from http://domain-a.com makes an <img> src request for http://domain-b.com/image.jpg. Many pages on the web today load resources like CSS stylesheets, images and scripts from separate domains.

For security reasons, browsers restrict cross-origin HTTP requests initiated from within scripts. For example, XMLHttpRequest and Fetch follow the same-origin policy. -MDN

What endpoints are supported?

Endpoints with a REST API can be managed with intREST.

Is intREST secure?

HTTPS, or HTTP secure, is the transport method utilized. This method of communication is widely used and trusted for transactions such as online banking. Your data is isolated to your system; specifically data is in your browser and managed endpoint while at rest and protected by TLS while in transit.

Troubleshooting

CORS issues cover 90% of problems. First use cURL to verify that your REST API is up and responding. Windows users can run cURL by downloading and installing Cygwin. Once validated with cURL, look at your browser console for errors. Managed devices must return the CORS header or you will have to install a CORS plugin on your browser. For Firefox users, CORS Everywhere works great. For Chrome users, Access Control Allow Origin * works.
